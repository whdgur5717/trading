에러 처리 흐름: 외부 실패에서 공개 API 응답까지

KIS, FSC, OpenDART에서 생긴 실패가 어디까지 내부 값으로 남고, 어느 지점에서 클라이언트가 받는 status/code/message로 바뀌는지 한 장으로 정리한 문서입니다.

핵심 결론 외부 API 실패 객체와 공개 API 에러 객체는 같은 것이 아닙니다. 외부 API 실패는 내부 진단값이고, 클라이언트 응답은 마지막 응답 포장 단계에서만 만들어집니다.

예시 하나로 보면 이렇게 바뀝니다

상황: 가격 API가 KIS 현재가를 가져오다가 실패

클라이언트는 GET /prices?symbol=005930를 호출했고, 우리 서버는 KIS 현재가 API를 호출했습니다. KIS가 503과 원문 메시지를 내려줬다고 가정합니다. 

KIS 응답

HTTP 503
{
  "msg_cd": "KIS999",
  "msg1": "KIS leaked token testAccessToken appsecret"
}

1. 외부 API 담당 코드는 내부 실패값만 만든다

이 단계에서는 아직 클라이언트 응답을 만들지 않습니다. KIS에서 무슨 일이 있었는지만 내부 값으로 남깁니다. 

return err({
  service: "kis",
  code: "unavailable",
  message: "KIS leaked token testAccessToken appsecret",
  endpoint: "/uapi/domestic-stock/v1/quotations/inquire-price",
  upstreamStatus: 503,
  upstreamCode: "KIS999",
})

2. 우리 서비스 에러로 바꿀 때 원문 메시지를 버린다

밖으로 나갈 에러 문구는 KIS 원문을 쓰지 않고, 우리가 정한 안전한 문구를 씁니다. 

case "unavailable":
  return {
    type: "market-data-unavailable",
    message:
      MARKET_DATA_ERRORS["market-data-unavailable"].message,
    details: {
      service: "kis",
      externalCode: "unavailable",
      upstreamStatus: 503,
      upstreamCode: "KIS999",
    },
  }

3. 응답 포장 코드가 status를 정한다

market-data-unavailable이면 클라이언트에게 502로 내려갑니다. 외부 API 담당 코드가 502를 직접 정하지 않습니다. 

API_ERROR_STATUS = {
  "market-data-unavailable": 502,
  "market-data-auth-unavailable": 502,
  "market-data-timeout": 504,
  "market-data-invalid-response": 502,
  "market-data-not-found": 404,
}

4. 클라이언트가 실제로 받는 값

KIS 원문 메시지에 token이나 appsecret 같은 문자열이 있어도 클라이언트 응답에는 들어가지 않습니다. 

HTTP 502
{
  "success": false,
  "error": {
    "status": 502,
    "code": "market-data-unavailable",
    "message": "Market data is unavailable",
    "details": {
      "service": "kis",
      "externalCode": "unavailable",
      "upstreamStatus": 503,
      "upstreamCode": "KIS999"
    }
  }
}

원칙

1

외부 API 담당 코드는 공개 응답을 만들지 않는다

KIS/FSC/OpenDART 어댑터 안에서는 ExternalServiceError<...> 모양의 실패값만 Result.err(...)에 담습니다. 여기서 HTTP status나 API envelope를 만들지 않습니다.

2

우리 서비스 에러로 한 번 바꾼다

가격, 수익률, 캔들 같은 서비스 로직이 받기 전에 외부 API 실패를 MarketDataError로 바꿉니다. 이때 외부 API 원문 메시지는 버리고, catalog에 있는 안전한 메시지만 사용합니다.

3

마지막에만 HTTP 응답으로 포장한다

ApiResponseInterceptor와 ApiFilter가 { success: false, error: ... }를 만듭니다. status도 여기서 API code 기준으로 결정됩니다.

전체 흐름

1. Request

Controller / Service

컨트롤러와 서비스는 성공값 또는 Result를 반환합니다. KIS 원문 status, 원문 message, 원문 body를 직접 판단하지 않습니다.

prices/prices.service.ts returns/returns.service.ts stocks/stocks.service.ts
2. External API

KIS / FSC / OpenDART

외부 응답을 읽고 실패 종류를 정합니다. 이 실패값은 아직 클라이언트에게 내려갈 에러가 아닙니다.

kis/response.ts fsc/fsc.response.ts opendart/opendart.response.ts
3. Internal Failure

ExternalServiceError

타입만 있는 내부 실패 구조입니다. 런타임 class가 아니며 new ExternalServiceError(...)를 만들지 않습니다.

common/error/externalService/error.ts common/error/externalService/code.ts
4. Service Error

MarketDataError

외부 API 실패를 우리 서비스에서 쓰는 에러로 바꿉니다. 메시지는 MARKET_DATA_ERRORS에서만 가져옵니다.

kis/error.ts fsc/fsc.error.ts opendart/opendart.error.ts
5. HTTP Response

Public Response

마지막 응답 포장 코드가 status, code, message, details를 가진 실패 응답으로 감쌉니다.

common/api/response.ts common/api/filter.ts common/api/api.errors.ts

외부 API별 실패 타입 범위

외부 API 내부 실패 타입 의미 변환 함수
KIS "unavailable" | "auth-unavailable" | "timeout" | "invalid-response" 시세 요청, 인증 실패, timeout, schema mismatch toMarketDataError
FSC response: "unavailable" | "auth-unavailable" | "invalid-response"
request: "unavailable" | "timeout" 		응답 파서와 요청 실패가 만들 수 있는 에러 범위를 분리 toFscMarketError
OpenDART "unavailable" | "auth-unavailable" | "timeout" | "invalid-response" | "not-found" OpenDART 응답 실패, 요청 실패, 필수 corp code 누락 toOpendartMarketError

공개 API status 결정 위치

invalid-request 400. 요청 query, path, body 검증 실패.
unsupported-stock 404. 형식은 맞지만 지원하지 않는 종목.
market-data-unavailable 502. 외부 시세 API의 일반 실패.
market-data-timeout 504. 외부 시세 요청 timeout.
market-data-not-found 404. 필수 시세 데이터가 없어 결과를 만들 수 없음.

수정된 구조

ExternalServiceError는 내부 외부 API 실패 타입으로만 존재합니다. 외부 API 담당 코드는 이 값을 Result.err(...)로 반환하고, market adapter의 공개 메서드에서만 MarketDataError로 바꿉니다.

API 응답은 ApiResponseInterceptor가 Result.err를 보고 만들고, 예외로 빠진 값은 ApiFilter가 같은 envelope로 감쌉니다.

밖으로 나가면 안 되는 것

외부 API 원문 message, raw body, cause, stack, token, secret, auth header, service key는 공개 응답에 넣지 않습니다. 공개 message는 catalog의 안전한 문구만 사용합니다.
